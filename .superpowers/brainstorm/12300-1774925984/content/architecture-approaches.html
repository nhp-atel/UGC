Agent Architecture Approaches

Three ways to structure the orchestrator + agent system. All use Temporal as the execution backbone.

A

Orchestrator as Temporal Workflow

The orchestrator IS the Temporal workflow. It calls agents as activities, evaluates results inline, and decides next steps using workflow logic (if/else, loops). Agents are activity functions with internal quality loops.

Architecture A — Workflow-as-Orchestrator
Trigger
  └─▶ Temporal Workflow (= Orchestrator)
        │
        ├─▶ callAgent("research", {topic, tone})
        │     └─ ResearchAgent.run() ──▶ OpenAI (loop until quality ✓)
        │
        ├─▶ evaluate(researchResult)
        │     └─ if quality < threshold → retry research
        │
        ├─▶ callAgent("creative", {quote, tone})
        │     └─ CreativeAgent.run() ──▶ OpenAI (loop until quality ✓)
        │
        ├─▶ evaluate(creativeResult)
        │     └─ if !cohesive → retry creative or re-research
        │
        ├─▶ callAgent("post", {quote, brief})
        │     └─ PostAgent.run() ──▶ OpenAI (loop until quality ✓)
        │
        ├─▶ finalReview(all outputs)
        │     └─ orchestrator LLM call: "is this cohesive?"
        │
        └─▶ saveDraft / publish

Pros

  • Simple — workflow IS the brain
  • Full Temporal durability for retries
  • Easy to debug (one workflow)

Cons

  • Orchestrator logic is hardcoded TypeScript
  • Adding new routing requires code changes
B

LLM-Driven Orchestrator Agent

A dedicated Director Agent (powered by LLM) decides what to do next. It receives context, picks which agent to call, evaluates results, and loops until satisfied. The Temporal workflow just executes what the Director decides.

Architecture B — LLM Director
Trigger
  └─▶ Temporal Workflow
        └─▶ DirectorAgent.run(input)     ◀── LLM-powered
              │
              ├─▶ thinks: "need quotes about discipline"
              │     └─ tool_call: research({topic: "discipline"})
              │           └─ ResearchAgent.run() ──▶ OpenAI
              │
              ├─▶ evaluates: "quote #2 is strongest"
              │     └─ tool_call: select({quoteIndex: 2})
              │
              ├─▶ thinks: "need cinematic brief for this"
              │     └─ tool_call: creative({quote, tone})
              │           └─ CreativeAgent.run() ──▶ OpenAI
              │
              ├─▶ evaluates: "hook is weak, retry"
              │     └─ tool_call: creative({quote, tone, feedback})
              │
              ├─▶ thinks: "now write the post"
              │     └─ tool_call: post({quote, brief})
              │
              └─▶ thinks: "all good, save it"
                    └─ tool_call: save({...})

Pros

  • Most truly "agentic" — LLM reasons about what to do
  • Can adapt to unexpected situations
  • Extensible via tools, not code rewrites

Cons

  • More expensive (extra LLM calls for routing)
  • Harder to predict behavior
  • More complex to debug
C

Hybrid — Structured Flow + LLM Quality Gates

The workflow defines the sequence (like A), but quality evaluation at each step is done by an LLM call (like B). The orchestrator doesn't freely choose what to do — it follows the pipeline but uses AI judgment at decision points.

Architecture C — Hybrid
Trigger
  └─▶ Temporal Workflow (fixed sequence)
        │
        ├─▶ ResearchAgent.run({topic})
        │     └─ internal loop: generate → evaluate → retry
        │
        ├─▶ QualityGate({research, context})    ◀── LLM judge
        │     └─ "pass" / "retry research" / "adjust params"
        │
        ├─▶ CreativeAgent.run({quote, tone})
        │     └─ internal loop: generate → evaluate → retry
        │
        ├─▶ QualityGate({creative, research})   ◀── LLM judge
        │     └─ "pass" / "retry creative" / "re-research"
        │
        ├─▶ PostAgent.run({quote, brief})
        │     └─ internal loop: generate → evaluate → retry
        │
        ├─▶ FinalReview({all outputs})          ◀── LLM judge
        │     └─ "publish" / "revise X"
        │
        └─▶ saveDraft / publish

Pros

  • Predictable flow, intelligent judgment
  • Best of both: structure + AI quality
  • Easier to debug than full LLM routing

Cons

  • Less flexible than full LLM orchestrator
  • Quality gates add latency + cost

Recommendation: Option B is the most genuinely agentic and aligns with your goal of an orchestrator agent. It's the architecture that best demonstrates AI agent patterns.
